# 欄位設計指南：為什麼要「動態依標題文字尋找欄位」

## 問題背景

最直覺的做法，是把 Google 表單/試算表的每一欄位置寫死成常數，例如：

```javascript
const COL = {
  GRADE: 2,  // B欄固定是年級
  CLASS: 3,  // C欄固定是班級
  ...
};
```

這在系統剛建好時完全沒問題，因為欄位順序就是照這樣建立的。但**問題出在系統上線之後**：
- 學校端（校長/老師）可能會自己在 Google 表單新增/修改題目
- Google 表單的規則是：**已連結試算表的表單，新增題目的資料一律會被加到「最後一欄」**，
  不會插入到原本欄位中間（詳見踩坑記錄坑3）
- 如果刪除某題後又重新建立「同名」題目，Google 會把它當成全新欄位，
  舊欄位停止收資料、新欄位又被加到最後面

如果欄位位置是寫死的常數，遇到上述任何一種情況，系統輕則資料讀取錯位、
重則整個排序/分流邏輯壞掉，而且很可能是「默默壞掉」（不報錯，但資料是錯的），
最難排查。

## 解決方案：動態依標題文字尋找

改成「執行時讀取試算表第一列的標題文字，建立『欄位代號 → 實際欄位編號』的對照表」：

```javascript
// 欄位代號 → 標題文字 對照表（這個是穩定的，不會變）
const HEADER_NAMES = {
  GRADE: '年級',
  CLASS: '班級',
  ...
};

// 核心必要欄位：找不到就要丟出清楚的錯誤，而不是默默算錯
const REQUIRED_HEADER_KEYS = ['GRADE', 'CLASS', 'SEAT', 'NAME', ...];

function getMainColMap_(sheet, throwOnMissing) {
  const lastCol = sheet.getLastColumn();
  const headerRow = sheet.getRange(1, 1, 1, lastCol).getValues()[0];
  const textToCol = {};
  headerRow.forEach((text, idx) => {
    const t = String(text || '').trim();
    if (t) textToCol[t] = idx + 1;
  });

  const map = {};
  Object.keys(HEADER_NAMES).forEach(key => {
    map[key] = textToCol[HEADER_NAMES[key]] || null;
  });
  map.LAST_COL = lastCol;

  if (throwOnMissing !== false) {
    const missing = REQUIRED_HEADER_KEYS.filter(key => !map[key]);
    if (missing.length) {
      throw new Error('找不到必要欄位標題：[' + missing.map(k => HEADER_NAMES[k]).join('、') + ']');
    }
  }
  return map;
}
```

**用法完全不變**：呼叫端還是寫 `COL.GRADE`、`r[COL.NAME - 1]`，跟原本寫死常數的
語法一模一樣，只是 `COL` 現在是「函式呼叫的結果」而不是「寫死的常數」：

```javascript
function syncDerivedSheets() {
  const mainSheet = ss.getSheetByName(SHEET_MAIN);
  const COL = getMainColMap_(mainSheet); // 唯一需要改的地方
  // 以下完全不用改
  const grade = r[COL.GRADE - 1];
  ...
}
```

這帶來幾個好處：
1. **新增表單題目完全安全**：新欄位不管被 Google 加到第幾欄，只要沒有跟現有的
   `HEADER_NAMES` 撞名，系統完全不受影響（多出來的欄位就是被忽略而已）
2. **核心欄位被誤刪會有明確錯誤**：不會默默算錯，會直接告訴使用者「缺少哪個欄位」
3. **初始化流程也能動態決定系統欄位要接在第幾欄**：不用假設表單一定有幾題，
   系統欄位永遠接在「當時試算表的最後一欄」

## 哪些欄位需要動態化，哪些不用

- **需要動態化**：所有直接來自 Google 表單、或使用者可能會在 UI 上手動調整順序的欄位
  （在原型專案裡是「彙整總表」的所有欄位，因為前 8 欄是表單直接寫入的）
- **不需要動態化**：完全由程式自己產生、使用者不會去動的工作表（例如各年級分流表、
  去識別化總覽），因為這些工作表的欄位順序是程式每次重建時自己決定的，
  不受表單異動影響，可以放心用寫死的欄位順序常數（例如 `GRADE_SHEET_FIELDS` 陣列）

## 部署前的驗證方式

牽動這麼多函式的重構，正式部署前建議寫一個 Node.js 腳本模擬 GAS 環境
（模擬 `SpreadsheetApp`/`FormApp`/`PropertiesService` 的最小行為），
實際跑過幾個關鍵情境：
1. 初始化流程（第一次執行，欄位位置是否正確接在表單欄位後面）
2. 正常的表單送出→同步流程
3. **刻意模擬「使用者在表單新增一題」**（新欄位被加到最後面），確認同步流程還能正常運作
4. **刻意刪掉某個核心欄位的標題文字**，確認會丟出清楚的錯誤而不是默默壞掉

全部驗證通過再交付部署，可以大幅提高對重構安全性的信心。
