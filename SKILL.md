# Skill：校園集體症狀通報系統 建置技能

> 原型專案：新北市忠義國小校園出現集體腸胃不適症狀事件通報系統（2026-07 開發完成並正式上線）。
> 本 skill 把整套開發流程、架構決策、以及開發過程中踩過的坑都整理成標準化流程，
> 讓任何電腦上的任何 AI 助理，讀完這份文件後都能協助使用者重建一套相同架構的系統。

**在開始任何開發動作之前，請先完整讀過以下三份文件：**
- `docs/建置手冊.md`：完整的步驟流程（表單→試算表→GAS Web App→GitHub Pages）
- `docs/踩坑記錄.md`：11 個已知的坑與對應解法，**開發中遇到類似錯誤請先查這份文件**
- `docs/欄位設計指南.md`：為什麼要用「動態依標題文字尋找欄位」而不是寫死欄位編號

`reference/Code.gs` 是一份**可直接執行的參考實作範例**（來自忠義國小的正式版本，已去除該校
專屬的表單ID、logo圖檔，並在需要客製化的地方標註 `TODO`），不是空白樣板，請以此為基礎修改。
`reference/github-pages-index.html` 同理，是 GitHub Pages 外殼的參考實作。

## 當使用者說「建立校園集體症狀通報系統」時，依序執行：

### 步驟 0：讀懂架構
讀取本 repo 的 `docs/建置手冊.md`、`docs/踩坑記錄.md`、`docs/欄位設計指南.md`，
完整理解整套架構（Google 表單只負責導師通報的初始欄位；試算表當簡易資料庫；
GAS Web App 提供首頁動態表格 + 三組通報角色的密碼登入搜尋更新介面 + 管理者後台；
GitHub Pages 只是包一層導覽列外殼，實際內容用 iframe 嵌入 GAS Web App）。

### 步驟 1：詢問 Google 表單規格（一次問清楚，避免日後修改）
請開發者準備好並提供：
1. 表單標題與說明文字
2. 每一題的題目內容
3. 每一題是否必填
4. 每一題的性質（下拉選單 / 簡答 / 核取方塊 / 單選鈕）以及選項內容

**務必提醒開發者**：表單題目請盡量一次規劃到位。雖然系統已經用「動態依標題文字尋找欄位」
（見 `docs/欄位設計指南.md`）大幅降低了事後修改的風險，**新增題目是安全的**，
但仍然強烈建議不要事後刪除或搬動核心欄位（年級/班級/座號/姓名/聯絡電話等）。

### 步驟 2：詢問學校識別資訊
1. 學校名稱
2. 學校 logo 圖檔（是否需要去背、尺寸調整；若使用者能提供圖檔路徑，
   AI 助理可用 Python Pillow 做去背+裁切+縮圖，再轉 base64 內嵌到首頁 HTML，
   做法可參考 `docs/建置手冊.md` 的「Logo 處理」章節）
3. 配色偏好（暖色系/冷色系/其他指定色系）

### 步驟 3：詢問導覽列功能按鈕規劃
參考實作預設的導覽列有：首頁 / 導師通報 / 檢傷組通報 / 休息組通報 / 送醫組通報 / 管理者登入。
詢問開發者：
- 是否全部保留？
- 是否要增加/減少角色（例如把「送醫組」拆成更細的角色、增加「衛生組」等）？
- 每多一個角色，代表要在 `GROUP_CONFIG` 常數多加一組設定（顏色、密碼屬性名、
  負責填寫的欄位），並在 `doGet()` 路由多一個判斷分支。

### 步驟 4：確認密碼設定方式
說明密碼一律存在 Apps Script「指令碼屬性」（`PropertiesService`），不會寫死在程式碼或前端，
確認密碼屬性的命名（預設 `PWD_TRIAGE`/`PWD_REST`/`PWD_HOSPITAL`/`PWD_ADMIN`，
可依步驟3決定的角色名稱調整）。**AI 助理不需要、也不應該知道實際密碼內容**，
請開發者自己到 Apps Script 後台設定。

### 步驟 5：依規格客製化程式碼
以 `reference/Code.gs` 為基礎，依照步驟1~4收集到的規格客製化：
- `HEADER_NAMES`／表單題目與欄位對照
- `createReportForm()` 裡的題目、選項內容
- `GROUP_CONFIG`（角色名稱、顏色、密碼屬性名稱、負責欄位）
- `OPTIONS`（各角色專屬下拉選單內容）
- 首頁 HTML 的校名、配色、（若有提供）logo base64
- `GRADE_LIST`、`gradeToNumber_()`（如果該校年級範圍不同，例如沒有幼兒園，或有國中部）

### 步驟 6：帶使用者依 `docs/建置手冊.md` 完成部署
包含：建立 Google 表單（只能執行一次）→ 初始化試算表結構（`setupSpreadsheet`，
可安全重複執行）→ 安裝觸發器（`installTriggers`）→ 部署 GAS Web App →
建立 GitHub Pages 外殼（如果要用 `<帳號>.github.io` 帳號首頁形式，
repo 名稱必須完全是這個格式，且必須是 public repo）。

### 步驟 7：主動排除問題
開發過程中遇到任何錯誤，**先查 `docs/踩坑記錄.md` 是否為已知問題**，
用清楚的步驟詳細協助開發者排除，不要讓開發者自己反覆摸索——
這是本 skill 存在的核心價值。

### 步驟 8：建議建立專案自己的進度文件
比照原型專案的模式，建議在新專案裡建立一份「專案進度.md」，記錄該校的技術決策與
新踩到的坑，方便日後其他人接手或這個 skill 未來的版本迭代。

## 核心原則（不可妥協的工程守則）

1. **密碼絕對不寫死在程式碼或前端 JS**，一律用 `PropertiesService`
2. **彙整總表一律用「動態依標題文字尋找欄位」**（`getMainColMap_`），不要走回頭路用寫死欄位編號
3. **任何程式碼異動都提供「完整版單一檔案」**給開發者複製貼上，不要片段修改，
   避免開發者自己手動拼接時出錯（開發者可能不是工程背景）
4. **牽動多個函式的重構，部署前先寫模擬測試驗證**（可以用 Node.js 模擬
   `SpreadsheetApp`/`FormApp` 等物件的最小行為，實際跑過幾個關鍵情境再交付）
5. **彙整總表是唯一正本資料**，其他工作表（各年級分頁、去識別化總覽）都是自動產生的檢視表
6. **去識別化規則**：姓氏+隱藏中間字（○）+保留最後一字，供首頁公開顯示使用，
   真實姓名/聯絡電話只出現在管理者後台跟彙整總表
